Terminlər lüğəti

Biznes analitika terminlərinin Azərbaycan dilində sadə izahı və real iş nümunələri. 6 kateqoriya üzrə 40+ termin.

← Ana səhifə

Heç bir termin tapılmadı. Başqa söz ilə yoxla.

📊 Business Analysis

Stakeholder

Maraqlı tərəf

Layihənin nəticəsində maraqlı olan və ya ona təsir edə bilən istənilən şəxs və ya qrup.

💼 Nümunə: Bank layihəsində müştərilər, IT komandası, hüquq şöbəsi və rəhbərlik — hamısı stakeholderdir.

Business Requirement

Biznes tələbi

Şirkətin biznes məqsədini ifadə edən yüksək səviyyəli tələb — yəni layihənin niyə görüldüyü.

💼 Nümunə: «Onlayn satışları 6 ayda 20% artırmaq» — bu bir biznes tələbidir.

Scope

Əhatə dairəsi

Layihəyə nəyin daxil olduğu və nəyin daxil olmadığı. Sərhədləri müəyyən edir.

💼 Nümunə: Tətbiq layihəsində «yalnız iOS, Android sonraya» deyə scope müəyyən edilir.

Gap Analysis

Boşluq analizi

Cari vəziyyət (as-is) ilə hədəf vəziyyət (to-be) arasındakı fərqin müəyyən edilməsi.

💼 Nümunə: Hesabatlar əl ilə hazırlanır, hədəf avtomatlaşdırmadır — boşluq «avtomatik hesabat sistemi yoxdur».

SWOT Analysis

SWOT analizi

Güclü tərəflər, zəif tərəflər, imkanlar və təhdidlərin qiymətləndirilməsi.

💼 Nümunə: Yeni məhsuldan əvvəl komanda SWOT cədvəli qurub bazara çıxış riskini qiymətləndirir.

As-Is / To-Be

Cari və hədəf vəziyyət

Prosesin indiki (as-is) və olması istənilən (to-be) halı. Dəyişikliyi planlamaq üçün müqayisə edilir.

💼 Nümunə: As-is: müştəri 5 forma doldurur; to-be: tək onlayn forma.

Use Case

İstifadə halı

İstifadəçinin sistemlə qarşılıqlı əlaqəsini addım-addım təsvir edən ssenari.

💼 Nümunə: «Kartla ödəniş» use case-i: istifadəçi kartı daxil edir → sistem təsdiqləyir → çek göndərilir.

📝 Requirements (Tələblər)

Functional Requirement

Funksional tələb

Sistemin nə etməli olduğunu təsvir edən tələb.

💼 Nümunə: «İstifadəçi şifrəsini email vasitəsilə bərpa edə bilməlidir.»

Non-Functional Requirement

Qeyri-funksional tələb

Sistemin necə işləməli olduğunu (sürət, təhlükəsizlik, etibarlılıq) təsvir edən tələb.

💼 Nümunə: «Səhifə 2 saniyədən az açılmalıdır.»

User Story

İstifadəçi hekayəsi

İstifadəçi baxımından yazılmış qısa tələb: «Bir [rol] olaraq, [məqsəd] istəyirəm ki, [fayda]».

💼 Nümunə: «Bir müştəri olaraq sifarişimi izləmək istəyirəm ki, nə vaxt çatacağını biləm.»

Acceptance Criteria

Qəbul meyarları

User story-nin «tamamlandı» sayılması üçün ödənməli olan konkret şərtlər.

💼 Nümunə: «İstifadəçi yanlış şifrə daxil edəndə xəta mesajı görünməlidir» — bir qəbul meyarıdır.

BRD

Biznes tələbləri sənədi

Business Requirements Document — layihənin biznes tələblərini toplayan rəsmi sənəd.

💼 Nümunə: Yeni CRM layihəsində BA bütün tələbləri BRD-də sənədləşdirib rəhbərliyə təqdim edir.

Elicitation

Tələblərin toplanması

Stakeholderlərdən tələbləri çıxarma prosesi — müsahibə, sorğu, müşahidə yolu ilə.

💼 Nümunə: BA satış komandası ilə müsahibə apararaq tələbləri elicit edir.

Traceability Matrix

İzlənmə matrisi

Hər tələbin dizayn, kod və testlə əlaqəsini izləyən cədvəl. Heç bir tələbin unudulmamasını təmin edir.

💼 Nümunə: Matris göstərir ki, «şifrə bərpası» tələbi hansı test case ilə yoxlanılıb.

🔄 Agile / Scrum

Sprint

Sprint

Adətən 1-4 həftəlik qısa iş dövrü; sonunda hazır məhsul hissəsi təhvil verilir.

💼 Nümunə: Komanda 2 həftəlik sprintdə «login səhifəsi» funksiyasını bitirir.

Backlog

İş siyahısı

Görüləcək işlərin prioritetlə düzülmüş siyahısı.

💼 Nümunə: Product backlog-da 50 user story var, ən vacibləri yuxarıda durur.

Product Owner

Məhsul sahibi

Backlog-u idarə edən və nəyin əvvəl görüləcəyini müəyyən edən rol.

💼 Nümunə: Product Owner müştəri rəyinə əsasən «ödəniş» funksiyasını birinci sıraya qoyur.

Scrum Master

Scrum Master

Komandanın Scrum prosesinə əməl etməsinə kömək edən və maneələri aradan qaldıran rol.

💼 Nümunə: Scrum Master komandanı bloklayan server problemini IT ilə həll edir.

Daily Standup

Gündəlik görüş

Komandanın hər gün keçirdiyi 15 dəqiqəlik qısa sinxronizasiya görüşü.

💼 Nümunə: Hər səhər hər kəs deyir: dünən nə etdim, bu gün nə edəcəm, nə məni bloklayır.

Velocity

Sürət (velocity)

Komandanın bir sprintdə tamamladığı işin ölçüsü (adətən story point ilə).

💼 Nümunə: Komandanın velocity-si sprintdə 30 story point-dir; planlaşdırma buna görə edilir.

Epic

Epik

Bir neçə user story-yə bölünən böyük iş bloku.

💼 Nümunə: «Ödəniş sistemi» epic-i kart ödənişi, qaytarma və çek kimi story-lərə bölünür.

Retrospective

Retrospektiv

Sprint sonunda komandanın nəyin yaxşı, nəyin pis getdiyini müzakirə etdiyi görüş.

💼 Nümunə: Retro-da komanda qərara gəlir ki, növbəti sprintdə testə daha çox vaxt ayırsın.

🔌 API və texniki terminlər

API

Proqram interfeysi

İki proqramın bir-biri ilə «danışmasını» və məlumat mübadiləsini təmin edən interfeys.

💼 Nümunə: Hava tətbiqi məlumatı hava xidmətinin API-si vasitəsilə alır.

Endpoint

Son nöqtə

API-də müraciət olunan konkret ünvan (URL).

💼 Nümunə: /users endpoint-i istifadəçilərin siyahısını qaytarır.

REST

REST

API-lərin qurulması üçün geniş istifadə olunan standart üsul.

💼 Nümunə: Çox veb xidmət REST API ilə HTTP üzərindən məlumat mübadiləsi edir.

JSON

JSON formatı

Məlumatın həm insan, həm maşın üçün oxunaqlı formatda mübadiləsi üçün istifadə olunan format.

💼 Nümunə: API cavabı {"ad": "Aytac", "yas": 25} — JSON formatındadır.

Database

Verilənlər bazası

Məlumatın strukturlu şəkildə saxlandığı sistem.

💼 Nümunə: Bütün müştəri məlumatları verilənlər bazasında cədvəllərdə saxlanılır.

SQL

SQL

Verilənlər bazasından məlumat sorğulamaq və idarə etmək üçün dil.

💼 Nümunə: BA «son aydakı sifarişlər»i görmək üçün sadə SQL sorğusu yazır.

Integration

İnteqrasiya

İki və ya daha çox sistemin birləşdirilərək birlikdə işləməsi.

💼 Nümunə: Onlayn mağazanın ödəniş sistemi ilə inteqrasiyası sayəsində kartla ödəniş işləyir.

🧪 Testing

Test Case

Test halı

Bir funksiyanı yoxlamaq üçün addımlar, giriş məlumatı və gözlənilən nəticə.

💼 Nümunə: Test case: «Boş email ilə qeydiyyat» → gözlənilən nəticə: xəta mesajı.

UAT

İstifadəçi qəbul testi

User Acceptance Testing — real istifadəçilərin sistemi təhvildən əvvəl yoxlaması.

💼 Nümunə: Müştəri komandası yeni sistemi UAT-da sınaqdan keçirib təsdiq edir.

Regression Testing

Reqressiya testi

Yeni dəyişikliyin köhnə, işləyən funksiyaları pozmadığını yoxlamaq.

💼 Nümunə: Yeni özəllik əlavə olunandan sonra login-in hələ də işlədiyini reqressiya testi yoxlayır.

Bug / Defect

Səhv (defekt)

Sistemin gözləniləndən fərqli və ya yanlış işləməsi.

💼 Nümunə: Düymə basıldıqda səhifə çökür — bu bug-dır, tester onu qeydə alır.

Smoke Testing

Smoke test

Sistemin əsas funksiyalarının ümumi işləkliyini sürətli yoxlamaq.

💼 Nümunə: Yeni versiya yüklənəndə smoke test ilə «açılır? login olur?» yoxlanılır.

Test Scenario

Test ssenarisi

Yoxlanılacaq funksiyanın yüksək səviyyəli təsviri; bir neçə test case-i əhatə edə bilər.

💼 Nümunə: «Ödəniş prosesini yoxla» ssenarisi kart, qaytarma və xəta hallarını əhatə edir.

QA

Keyfiyyətin təminatı

Quality Assurance — məhsulun keyfiyyətini təmin edən proses və komanda.

💼 Nümunə: QA komandası məhsul buraxılmadan əvvəl bütün testləri keçirir.

🚀 Product Management

MVP

Minimal işlək məhsul

Minimum Viable Product — ən az funksiya ilə bazara çıxarılan, dəyər verən ilk versiya.

💼 Nümunə: Çatdırılma tətbiqinin MVP-si yalnız sifariş və ödəniş edir; reytinq sonra əlavə olunur.

Roadmap

Yol xəritəsi

Məhsulun zamanla hansı istiqamətdə inkişaf edəcəyini göstərən plan.

💼 Nümunə: Roadmap göstərir ki, bu rüb mobil tətbiq, gələn rüb çoxdilli dəstək gələcək.

KPI

Əsas performans göstəricisi

Key Performance Indicator — uğuru ölçən konkret rəqəmsal göstərici.

💼 Nümunə: «Aylıq aktiv istifadəçi sayı» tətbiqin əsas KPI-dır.

Persona

İstifadəçi obrazı

Tipik istifadəçini təmsil edən uydurma, lakin real məlumata əsaslanan profil.

💼 Nümunə: «Aysel, 28 yaş, məşğul ana, sürətli alış-veriş istəyir» — bir persona.

Wireframe

Sxematik maket

Səhifənin sadə, detalsız (rəngsiz) strukturunu göstərən eskiz.

💼 Nümunə: BA dizaynerə düymə və mətnlərin yerini göstərmək üçün wireframe çəkir.

A/B Testing

A/B testi

İki variantı müqayisə edərək hansının daha yaxşı işlədiyini ölçmək.

💼 Nümunə: Qırmızı və yaşıl «Al» düyməsi A/B testi ilə sınanır — yaşıl daha çox klik alır.

Prioritization (MoSCoW)

Prioritetləşdirmə

Tələbləri Must / Should / Could / Won't kateqoriyalarına bölərək vaciblik sırası vermək.

💼 Nümunə: «Login — Must, qaranlıq rejim — Could» — MoSCoW ilə prioritet verilir.
